# 项目上下文：app/pkg

## 概览

本目录 (`app/pkg`) 存放了整个 `bs_server` 项目可复用的通用代码包。这些包与具体的业务逻辑解耦，提供了诸如配置管理、数据访问、图像处理、认证等基础能力。

## 在哪里寻找

| 包名          | 路径      | 职责                                                                         |
| ------------- | --------- | ---------------------------------------------------------------------------- |
| **GoCV 工具** | `gocv/`   | **[高风险]** 封装所有 OpenCV 图像处理功能。**必须严格遵守其内存管理约定！**  |
| **应用配置**  | `conf/`   | 使用 Viper 加载 `application.yaml` 文件，管理全局配置。                      |
| **数据访问**  | `data/`   | 提供对 MongoDB (`imongo`) 和 Redis (`iredis`) 的客户端封装和连接管理。       |
| **统一错误**  | `bsvo/`   | 定义了项目核心的 `AppError` 结构体和相关错误常量，是标准错误处理的基础。     |
| **JWT 认证**  | `bsjwt/`  | 封装了 JWT 的生成、解析和验证逻辑。                                          |
| **通用工具**  | `util/`   | 包含各类通用工具，如 Excel 处理 (`iexcel`)、Viper 辅助函数 (`viperutl`) 等。 |
| **响应辅助**  | `helper/` | 提供了 Gin 框架的响应辅助函数，用于快速生成成功的 JSON 响应。                |

## 关键包详解

### `gocv` - 图像处理 (高风险区域)

这是项目中技术风险最高的包，因为它直接调用 CGO 绑定了 OpenCV。

- **核心职责**: 提供图像的读取、解码、特征提取（如 SIFT, ORB）、变换和匹配等所有底层能力。
- **强制约定**: **任何与 `gocv.Mat` 对象的交互，都必须遵循“谁创建，谁释放”的原则**。创建后必须立即使用 `defer mat.Close()` 来防止内存泄漏。这是项目的最高开发约定之一，违反此约定将导致严重的性能问题和程序崩溃。
- **查找位置**:
    - `gocv/imgpro/`: 核心图像处理流水线和操作。
    - `gocv/imgpro/core/ops`: 具体的图像操作实现（如模糊、特征提取）。

### `conf` - 配置管理

- **核心职责**: 初始化并管理全局配置实例 (Viper)。
- **工作模式**: 在应用启动时 (`app/main.go`)，`conf.InitConfig()` 会被调用，它会读取 `conf/application.yaml` 文件，并将配置解析到 `conf.AppConfig` 结构体中。
- **修改配置**: 若要新增配置项，需要同时修改 `application.yaml` 和 `model/config.go` 中的结构体定义。

### `bsvo` - 值对象与错误

- **核心职责**: 定义 `AppError`，这是整个应用错误处理的基石。
- **使用方法**: 当 `internal/service` 或 `internal/handler` 中发生业务错误时，应创建并返回一个 `*bsvo.AppError` 实例。Gin 的错误处理中间件会自动捕获它并生成标准的 JSON 错误响应。
