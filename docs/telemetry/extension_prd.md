# OpenTelemetry 全链路监测扩展需求文档 (PRD)

## 1. 背景与目标

目前 `bs_server` 已完成基础的 OTel 初始化、Gin 框架及 MongoDB 的集成。然而，系统的核心业务——**图像处理流水线 (Image Processing Pipeline)** 和 **Redis 缓存层** 仍处于监测盲区，导致在分析识别延迟或缓存穿透问题时缺乏数据支撑。

**核心目标：**

- **消除黑盒**：将图像识别流水线的每一个步骤（如滤波、特征提取）转化为可视化的 Span。
- **监控缓存**：实时观测 Redis 的响应延迟及命令执行情况。
- **业务关联**：在 Service 层建立明确的业务跨度，提升链路追踪的可读性。

## 2. 详细需求

### 2.1 Redis 监测集成

- **需求描述**：自动记录所有 Redis 命令（GET, SET, LPUSH 等）的耗时、Key 信息（脱敏）及异常。
- **验收标准**：在 Trace 视图中能看到 `redis` 节点的调用，包含具体的命令名称。

### 2.2 图像处理流水线 (Image Pipeline) 深度追踪

- **需求描述**：对 `ImageProcessor` 的处理逻辑进行埋点。
- **关键指标**：记录每个 `Processor`（如 SIFT, Blur, Match）的 `Process` 方法耗时。
- **上下文穿透**：必须确保 `trace_id` 能从 API 层穿透到最底层的图像处理函数。

### 2.3 Service 层业务埋点

- **需求描述**：在关键业务逻辑（如 `ButterflyService` 的识别入口）手动开启 Span。
- **验收标准**：链路图不仅显示底层数据库/缓存耗时，还应清晰显示业务逻辑处理的耗时区间。

## 3. 非功能性需求

- **性能损耗**：在禁用 OTel 时，新增的埋点代码应具有极低的性能开销。
- **采样策略**：兼容现有的 OTel 采样策略，避免产生过量数据。
